ABOUT // 01 // UNDER THE HOOD

How it works

One local model. A small set of tools. You keep the final say.

ZyleBot is a FastAPI app between your browser and LM Studio. It stores conversations, sends the relevant context to the loaded model, runs permitted tools, and streams the result back. There is no hidden cloud service behind the chat.

Local inferenceExplicit toolsSQLite memoryHuman approval

ONE MESSAGE, END TO END

The path through ZyleBot.

  1. You send a messageThe browser posts your text—and an image when attached—to the local FastAPI server.
  2. ZyleBot builds the contextIt loads that conversation's saved history and sends it to the model running in LM Studio.
  3. The model answers or requests a toolTool calls are structured requests. The model does not receive unrestricted Python or filesystem access.
  4. The loop checks the requestRead-only tools can run automatically. Actions that can change the system pause for approval.
  5. The result returns to the modelIt can use the evidence, call another tool, correct course, or produce the final answer.
  6. The browser receives a live streamReasoning, tool events, approval cards, and answer text appear as the backend emits them.

WHAT RUNS WHERE

Your machine does the work.

LM Studio hosts the language model. FastAPI owns the agent loop and API. The browser is the interface. SQLite keeps conversations and messages. Voice transcription uses faster-whisper locally on the CPU.

Web search, page fetching, geocoding, and weather checks contact their respective internet services when you use those features. “Local” describes the model, app, and stored chat data—not an imaginary claim that every feature works offline.

WHAT THE MODEL CAN DO

Capabilities come from registered tools.

The model can ask ZyleBot to inspect files and system information, search or read the web, check weather, write files, create directories, and run shell commands. ZyleBot validates the tool name and arguments before dispatching anything.

A model response is still a prediction, not proof. Tools let it check reality; they do not make every conclusion correct.

CONTROL BOUNDARY

Reading can proceed. Impact needs permission.

Read-only tools normally run without interruption. File writes and directory creation proposed by the model stop the loop and show their exact arguments. Shell commands are classified first: known read-only commands can run automatically, dangerous commands are blocked, and everything else pauses for approval. Approval runs that call once; denial runs nothing and returns the refusal to the model.

The classifier and approval gate are not a sandbox. An approved command runs with the ZyleBot process's permissions. Read it as carefully as a command you typed yourself. See the Approval Gate for the full boundary.

MEMORY

The app remembers; the model does not.

Each conversation is stored separately in SQLite. On every turn, ZyleBot reconstructs the context from saved messages. A new chat starts clean; reopening one restores its history. The header gauge shows its estimated use of the loaded context window.

LIMITS

Local does not mean infallible.

Models can misunderstand, tools can fail, websites can block automated reading, and a finite context window cannot hold unlimited history. Image turns currently run without tools because the local stack does not reliably support both together.

The model proposes. Tools connect it to facts. ZyleBot enforces the boundary. You remain responsible for the decision.
