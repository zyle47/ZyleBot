PRODUCT // 04 // HUMAN AUTHORITY

Approval Gate

The model can propose the action. Only you can authorize it.

ZyleBot moves freely when it is gathering read-only information. The moment it wants to write a file, create a directory, append content, or invoke your shell, the Agent Loop stops. You see the exact proposal and decide whether that single action deserves your trust.

Exact arguments Approve or deny Persistent pause No silent execution

WHY A GATE MATTERS

Useful agency needs a boundary you can actually see.

A local agent has access to something more personal than a cloud account: your real files and your real shell, running with your user permissions. That power is what lets an assistant move from advice to action. It is also exactly why “the model seemed confident” is not a security policy.

The Approval Gate creates a clear transfer of authority. ZyleBot may reason about an action and prepare its arguments, but preparation is not permission. Until you approve, the action has not happened.

THE EXACT SEQUENCE

A real pause inside the Agent Loop.

  1. The model requests an action tool It chooses a registered CONFIRM_REQUIRED capability and supplies structured arguments.
  2. ZyleBot persists the pending state The assistant message and proposed tool call are stored before control returns to the interface.
  3. The approval card appears You see the tool name and arguments that would be passed to it — paths, content, command, or directory.
  4. The loop waits No action result is fabricated and no background execution races ahead while you are deciding.
  5. Your decision resumes the turn Approval executes that proposal once; denial returns a refusal result without executing it.

This is not a cosmetic modal over an already-running command. The backend marks the conversation as pending confirmation and the loop does not continue until the confirmation endpoint receives your decision.

TWO RISK TIERS

Routine observation stays fluid. Consequential action slows down.

SAFE // AUTO

Read-only tools

Listing directories, reading files, finding paths, inspecting system information, searching the web, fetching pages, and checking weather can run automatically.

They gather information without intentionally changing local state.
CONFIRM_REQUIRED // PAUSE

Action tools

Writing or appending files, creating directories, and executing shell commands always stop at the gate when proposed by the model.

They can change the system, so they require your explicit decision.

The tier belongs to the tool definition, not to the model's opinion of the moment. A persuasive explanation cannot downgrade run_command to safe, and a different loaded model cannot opt out of the registry's policy.

APPROVE

Execute this proposal once.

Approval authorizes the pending call shown on the card. ZyleBot invokes it, stores the result, and gives that result back to the model. The model can then explain what happened, verify the outcome with safe tools, or propose a separate next action.

A later action is a new decision. Approving one write does not create blanket permission for every write the model might imagine afterward.

DENY

Continue without executing it.

Denial does not kill the conversation or pretend the request succeeded. The proposed call is refused, the loop resumes, and the model receives that refusal as the tool result.

It can explain why the action was useful, suggest a safer alternative, or finish with instructions you can carry out yourself. “No” remains a productive answer.

WHAT YOU CAN INSPECT

Approval is attached to concrete arguments, not vague intent.

write_fileThe destination path and the full text proposed for the file.
append_fileThe target path and the exact content that would be added.
make_directoryThe precise directory path the tool would attempt to create.
run_commandThe command string proposed for your native shell.

Read carefully, especially when shell syntax can expand wildcards, interpolate variables, or affect more than one file. The gate exposes the proposal; it does not guarantee that the model understood every consequence of the command it wrote.

AN HONEST SECURITY BOUNDARY

The gate is a decision point, not a sandbox.

When you approve run_command, the command runs with the privileges of the ZyleBot process. It can be as powerful as something you typed into that shell yourself. Approval does not make an unsafe command safe, and it does not contain the command after execution begins.

Treat the card like a terminal prompt from a very fast collaborator: check the path, read the command, understand the scope, and deny anything you would not run manually. The protection is that the model cannot cross this boundary silently — judgment remains yours.

SURVIVES NAVIGATION

A refresh does not turn waiting into permission.

Pending confirmation is stored with the conversation, not held only in the browser's memory. Close the tab, refresh the page, or reopen the conversation and the approval card is rebuilt from that saved state. The loop is still waiting where you left it.

This persistence matters because a safety boundary should not depend on keeping one JavaScript object alive. Reloading can recover the interface; it cannot skip the gate.

EXAMPLE // FILE CHANGE

“Create a configuration file.”

ZyleBot can inspect nearby examples and draft the content with safe tools. When it requests write_file, you inspect the destination and proposed configuration. Approve if both are right; deny if it targeted the wrong place or chose values you do not want.

EXAMPLE // COMMAND

“Run the relevant verification.”

The Agent Loop can find the project's test command, but invoking it still raises a card. You can confirm that the command is read-only and correctly scoped before allowing it to consume time, create artifacts, or execute project code on your machine.

DIRECT USER CONTROLS

Not every button in the app is a model-proposed tool call.

The Approval Gate governs action tools selected by the Agent Loop. When you directly choose a model from the header or press Start server, your click is already the initiating human action, so ZyleBot does not place a second confirmation card in front of it.

The distinction is intentional: proposals originating from the model require a transfer of authority; explicit controls operated by you are already authorized by the interaction itself.
