PRODUCT // 02 // CAPABILITIES

Tool Arsenal

A model can know a lot. Tools let it find out what is true right now.

ZyleBot can move beyond its training data to inspect your workspace, read live system state, research the web, and — when you explicitly allow it — take action. Every capability is declared, typed, and placed in a clear risk tier before the model ever sees it.

WHY TOOLS CHANGE THE ANSWER

Fluency is useful. Evidence is better.

Without tools, a language model can only respond from the context you paste and the patterns it learned in the past. It might explain how to inspect a folder, suggest a search query, or tell you which command to run. With tools, it can inspect the folder, perform the search, and bring the result back into the conversation.

That is the point of the Arsenal: not a giant list of integrations for a feature table, but a focused set of capabilities that close the gap between “here is what you could do” and “here is what I found.”

SAFE TIER // READ-ONLY

Fast access to facts that cannot change your machine.

FS.01

Navigate

list_directory reveals what is inside a folder so the model can orient itself before reading.

FS.02

Read

read_file opens text content with pagination for material too large to consume in one pass.

FS.03

Discover

search_files finds matching paths recursively without making the model guess where something lives.

FS.04

Inspect

get_file_info returns file and directory metadata without opening or modifying the target.

SYS.01

Check the machine

get_system_info reports operating system, CPU, memory, disk, and other useful local state.

WX.01

Check the weather

get_weather gets current conditions through Open-Meteo instead of asking the model to improvise a forecast.

These tools are classified as SAFE: they retrieve information but do not write files, create folders, or execute arbitrary commands. The Agent Loop can call them without interrupting you for routine approval.

WEB.01

Search for the right source.

web_search uses DuckDuckGo to discover current pages without requiring an API key. It is the scout: useful for finding the exact article, documentation page, or source that should be read next.

Search snippets are leads, not final evidence. The Agent Loop is instructed to follow a promising result into the page whenever the question needs more than a headline.

WEB.02

Read beyond the snippet.

fetch_url downloads a page and extracts readable text. Long pages are returned in chunks with an offset, allowing the loop to continue through a document instead of silently truncating it and pretending the missing half did not matter.

Some bot-protected or heavily scripted sites still refuse automated reading. When that happens, the error becomes part of the loop and ZyleBot can fall back to search results or another source. The Arsenal extends reach; it does not fake access.

CONFIRM_REQUIRED // ACTIONS

When a tool can leave a footprint, you get the final word.

write_fileCreate or replace a text file with content you can inspect before approval.
append_fileAdd text to an existing file without replacing what is already there.
make_directoryCreate a directory at the exact path proposed by the model.
run_commandInvoke the native shell — PowerShell on Windows, the configured shell on macOS or Linux.

These are intentionally powerful and intentionally slower. The loop pauses, displays the tool name and arguments, and waits. Nothing executes until you approve that specific proposal. The Approval Gate is what makes action a deliberate collaboration instead of a hidden side effect.

COMPOSITION

The useful part is how tools work together.

Research a changing topic web_search → choose a source → fetch_url → read more if truncated → synthesize
Understand an unfamiliar project list_directory → search_files → read_file → explain the structure
Prepare a local change inspect files → reason about the edit → propose write_file → wait for approval → verify
Diagnose a machine get_system_info → interpret the state → propose a command only if deeper inspection is worth it

Each result returns to the Agent Loop. The model is not following a hard-coded macro; it chooses the next available capability based on what the last one revealed. That is how a small toolkit supports a much larger range of work.

ONE REGISTRY

Capabilities stay consistent across models.

Tools are defined once in a central registry with a name, description, input schema, implementation, and risk tier. ZyleBot turns that registry into the tool schema offered to the currently loaded model.

Switching models through Model Control changes the brain using the tools, not the tools themselves. A stronger tool-calling model may plan better, but it receives the same declared boundaries.

CONTROLLED OUTPUT

Enough information, without an accidental flood.

File reads, fetched pages, command output, and other potentially large results have configurable limits. Paginated tools tell the model how to continue when more content is available; command execution also has time and output ceilings.

These limits protect the local context window and keep one noisy tool from swallowing the whole conversation. More data is available when the task needs it, but it arrives in deliberate pieces.

DESIGNED FOR LOCAL TRUST

No mystery capability hiding behind the chat box.

ZyleBot can only call tools that the application explicitly registers and exposes. A model cannot invent a new privileged operation and make the backend obey it. If a tool is absent, it is unavailable. If it is marked for confirmation, it goes through the gate.

That makes the Arsenal understandable enough to audit: a short, concrete list of ways the model can observe the world and an even shorter list of ways it can change it.
