THE LOCAL ADVANTAGE

The model is a component you control, not an account you rent.

ZyleBot talks to LM Studio's OpenAI-compatible server on your machine. Your chat history lives in the app's local database, inference happens on hardware you can see, and changing the model does not require rewriting the application around a new cloud provider.

Local models come with a different kind of responsibility: you choose what is loaded, how much context it receives, and how those choices fit your hardware. Model Control keeps those decisions close to the conversation instead of burying them in a terminal window.

SWITCHING MODELS

Change the brain without changing the workspace.

  1. Choose an alias Select a friendly model name from the header instead of copying a long repository identifier.
  2. Unload the current model ZyleBot asks the lms CLI to free the active runtime before loading the replacement.
  3. Load the selected model The alias resolves to its actual LM Studio model id and configured context length.
  4. Continue the conversation The chat stays in place; the next inference is handled by the newly loaded engine.

There is no server restart and no manual configuration edit in the middle of your work. The operation is a direct choice you made in the interface, so it does not go through the tool Approval Gate used for model-proposed actions.

CONFIGURATION

Friendly names on top, exact identities underneath.

models.json is the source of truth for model aliases, real LM Studio ids, and context limits. The interface can say something memorable while the backend resolves the precise model artifact that LM Studio expects.

Keeping this mapping in one small file matters because downloaded model ids can be long and may change when multiple quantizations are installed. Update the mapping once and the rest of ZyleBot keeps speaking the same language.

MODEL-AGNOSTIC CORE

Same product, different strengths.

The conversation system, SQLite memory, interface, and Tool Arsenal do not change when the model changes. What changes is the model's reasoning style, speed, language ability, vision support, and skill at selecting and formatting tool calls.

That separation lets you experiment honestly. A fast small model can handle routine chat; a more capable model can take the difficult turn. ZyleBot remains the stable operating layer around both.

THE CONTEXT GAUGE

A fuel gauge for the model's working memory.

Every message, tool call, tool result, and system instruction sent to the model consumes part of its context window. Long conversations can feel infinite in the interface while the model's usable working space is anything but. The header makes that hidden constraint visible.

0–69% // comfortable
70–89% // watch the window
90%+ // high pressure

The gauge uses actual usage reported for the active conversation and the configured window for the loaded model. It turns amber at 70 percent and red at 90 percent. Routine health polling preserves the last known usage instead of making the bar flicker back to zero between turns.

RECOVERY

If LM Studio is down, the interface says so.

ZyleBot polls the local server and reports an offline state instead of leaving a dead chat box to explain itself. When the server is unavailable, a Start server control appears and runs lms server start for you.

Once LM Studio responds, ZyleBot synchronizes the active model state and returns the chat controls to service. You still need LM Studio installed and a model downloaded; the button removes ceremony, not prerequisites.

HARDWARE REALITY

Local control means honest tradeoffs.

Larger models and wider contexts ask more of VRAM and system memory. More capacity can buy stronger reasoning or longer continuity, but it can also slow inference dramatically when the workload spills beyond the GPU.

Model Control does not pretend every model fits every machine. It gives you the visibility to find the useful balance: quality you trust, speed you can tolerate, and a context window that stays inside the hardware budget you actually own.

WHAT SWITCHING DOES — AND DOES NOT — DO

The engine changes. Your local workspace remains yours.

  • It does route future model responses through the newly selected local model.
  • It does update the active context limit used by the interface gauge.
  • It does not delete or fork the current conversation.
  • It does not change which application tools are registered.
  • It does not send your stored chat history to a hosted ZyleBot service.

The next model receives the conversation context ZyleBot sends for the next turn. That makes switching powerful, but it also means a model with a smaller window or weaker tool support may interpret the same workspace differently. You are changing the reasoner, not rewriting the past.
